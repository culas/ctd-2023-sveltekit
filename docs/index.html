History, Origin

Timeline

  • 2016: first release of Svelte by Rich Harris
  • 2019: version 3 brought large changes
  • 2020: SvelteKit announced as replacement for Sapper
  • 2022: Rich Harris gets hired by Vercel to work on Svelte
  • 2022: official SvelteKit 1.0 release in December
  • version 3: leaner syntax, today's reactivity system
  • Sapper was the initial app framework for Svelte and the predecessor of SvelteKit
  • SvelteKit was release after a long beta phase to make sure "they get it right"

Concepts

Key Concepts of Svelte

  • compiled to optimised vanilla JS, no runtime library
  • single-file components, with scoped CSS
  • syntax close to the native web (HTML, CSS, JS)
  • easy reactivity and state management
  • little to no boilerplate code

Their own slogan is Â«cybernetically enhanced web appsÂ».

For me, learning and writing Svelte apps has made me more excited about web development than any other technology since I discovered JS all these years ago. It truly feels like you're just writing for the web and not for a specific framework.

Syntax

Svelte is a superset of the Â«webÂ»:

  • extends HTML by adding JS expressions (curly braces)
  • extends CSS by adding scoping mechanisms
  • extends JS by making reactivity a language primitive

Fun fact: an empty *.svelte file is already a valid Svelte component.

The syntax is essentially plain HTML, CSS and JavaScript with a little bit of extras on top.

Regarding the fun fact: The filename defines the component tag name.

Technically, one could argue:

Svelte is a language, not a framework

Rich Harris did exactly that in a blog post. Essentially: Svelte extends the languages of the web and has a compiler. That's why it feels so natural to write apps with Svelte.

SvelteKit

Features

  • mix and matching rendering methods for each page
  • accessible client-side routing with automatic preloading
  • filesystem-based routes with nested layouts
  • environment variable handling
  • type-safe data loading
  • built-in form actions that work with or without JS
  • automatic CSRF protection
  • can be used for libraries as well as web apps
  • deploy anywhere with adapters

Rendering methods:

  • SSG (static site generation, prerendered)
  • SSR (server side rendering, default)
  • CSR (client side rendering, SPA mode)
